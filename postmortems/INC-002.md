


# INC-002: Disk Exhaustion on NovaPay Payment Server

**Date:** 2026-06-02  
**Severity:** P1 - Critical  
**Status:** Resolved  
**Duration:** ~15 minutes  
**On-call Engineer:** Chirag Venkateshaiah  

## Summary
Disk usage on the NovaPay production server climbed from 50% to 80% due to 
a large temporary file written to /tmp. The payment API remained healthy 
during the incident due to the disk not reaching critical threshold before 
resolution. Identified and resolved within 15 minutes.


## Timeline
| Time (UTC) | Event |
|---|---|
| 04:10:00 | Baseline established — disk at 50%, service healthy |
| 04:17:00 | dd command wrote 2GB file to /tmp — disk climbed to 80% |
| 04:18:00 | watch -n 2 df -h confirmed live disk spike |
| 04:18:30 | du -sh /* run — /tmp identified as culprit |
| 04:19:00 | Permission error on du revealed need for sudo |
| 04:19:30 | sudo du confirmed 2.1G in /tmp/bigfile |
| 04:20:00 | rm /tmp/bigfile executed — disk returned to 50% |
| 04:20:30 | Service health verified — no payment impact |


## Root Cause
A 2GB zero-filled file was written to /tmp simulating runaway log growth or an uncontrolled file write from an application process. The file consumed 2GB of the 6.8GB root filesystem pushing usage to 80%

## Impact
- NovaPay Payment API: No impact - disk did not reach critical threshold
- Payments: No failed transactions during this incident
- Risk: Had disk reached 95%, systemd journal writes would fail, app logs would be lost, and payment writes would start failing


## Diagnostic Commands Used
| Command | Purpose | What It Reads |
|---|---|---|
| df -h | Filesystem usage overview | Filesystem superblock |
| du -sh /* | Directory size breakdown | Individual file inodes |
| sudo du -sh /tmp/ | Accurate /tmp size with permissions | File inodes with root access |
| free -m | Memory and swap status | /proc/meminfo |
| watch -n 2 df -h | Live disk monitoring during incident | Filesystem superblock every 2s |
| script -a | Full terminal session capture | Pseudoterminal (pty) |


## Key Learnings
1. df and du can disagree - df reads the superblock instantly, du walks inodes and requires correct permissions to be accurate
2. Always run du with sudo during incidents to avoid silent permission errors causing under-reporting
3. watch -n 2 is essential during disk incidents for live monitoring
4. script should be started immediately on paging - captures full incident timeline automatically
5. Zero swap means no buffer - when memory is exhausted processes are killed immediately with no warning

## What Went Well
- Baseline metrics established before incident simulation
- Live monitoring via watch caught the spike in real time
- Root cause identified within 2 minutes using du
- Terminal session captured via script for postmortem evidence
- Service remained healthy throughout - fast response prevented impact

## What Went Wrong
- Initial du run without sudo under-reported /tmp size
- No automated disk alert configured - relied on manual monitoring
- No runbook existed for disk exhausted incidents
