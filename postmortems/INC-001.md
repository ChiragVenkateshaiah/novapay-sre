


# INC-001: SSH Disconnection During Payment Service Verification

**Date:** 2026-06-01
**Severity:** P3 - Low
**Status:** Resolved
**Duration:** ~10 minutes
**On-call Engineer:** Chirag Venkateshaiah

## Summary
SSH connection dropped during active incident verification of the NovaPay Payment API. The payment service remained unaffected due to systemd service management.One test Payment was successfully processed before disconnection.

## Timeline
| Time (UTC) | Event |
|----|-----|
| 04:00:55 | NovaPay service started via systemd |
| 04:05:05 | Health check verified — service healthy |
| 04:16:18 | Second health check during investigation |
| 04:17:15 | Test payment PAY-22403 processed successfully |
| ~04:18:00 | SSH session dropped — client_loop: send disconnect: Broken pipe |
| 04:33:00 | SSH reconnected — service confirmed still running |

## Root Cause
SSH connection timeout due to idle network session between WSL2 client and EC2 instance. No keepalive configuration was set on the SSH connection.

## Impact
- NovaPay Payment API: No impact - service continued running
- Payments: No failed transactions
- Engineers: ~15 minutes of lost terminal session

## What Went Well
- systemd service management kept NovaPay running after SSH disconnection
- Service self-recovered without manual intervention
- Logs preserved full timeline of events

## What Went Wrong
- No SSH keepalive configured - connection dropped silently
- No tumux session active - terminal state was lost on disconnect

## Action Items
| Action | Owner | Priority |
|---|---|---|
| Configure SSH keepalive on WSL2 | Chirag | High |
| Always work inside tmux on EC2 | Chirag | High |
| Add payment confirmation acknowledgment to app | Chirag | Medium |

## Lesson Learned
1. systemd is the correct way to run application services - SSH lifecycle should never affect service lifecycle
2. SSH keepalive prevents silent disconnections
3. Payment success status should reflect confirmed receipt, not just local processing
