## Core concept — Documentation as a system property

### The concept explained from first principles

In most projects, documentation is afterthought — written after the system works, describing what the code already does, living in a separate place from the decisions that shaped it. This produces documentation that drifts from reality (the code changes, the docs don't), and documentation that can't answer "why" (it describes what, not the reasoning behind it).

A different model treats documentation as a system property: it is part of the system's correctness guarantee, maintained with the same discipline as the code, and structured to answer the questions a future operator will actually ask.

For NovaPay, this means three layers:

**Layer 1 — The README as evidence, not description.** A portfolio README is not a tutorial and not a feature list. It is a case study: here is what was built, here is what was observed, here is what was hardened, and here is the evidence (commit hashes, incident numbers, invariant query results). A reader should be able to reconstruct what happened from the README alone, then go into the day notes or ADRs for depth.

**Layer 2 — ADRs as frozen decisions.** An Architecture Decision Record captures a decision at the moment it was made: the context (what problem existed), the decision (what was chosen), the alternatives (what was considered and rejected), and the consequences (what this commits to, what it forecloses). Critically: ADRs are immutable. They are not updated when the system changes; instead, a new ADR is written that supersedes the old one. This preserves the reasoning history. ADR-004 (full jitter retry) documents why AWS research favoured full jitter over equal jitter — that reasoning is still valid even if the retry logic changes later.

**Layer 3 — CLAUDE.md as session context.** The CLAUDE.md file is read by Claude Code at the start of every session. It encodes the project's state, constraints, and coding conventions so that an AI coding assistant doesn't need to re-derive context from the codebase on every session. It is a form of documentation that serves a machine reader rather than a human reader — structured differently (explicit rules, not prose), but with the same goal: transfer knowledge across a session boundary without loss.

### Why it matters for a payments service specifically

A payments service that nobody can explain is a liability. When a charge fails at 3am, the engineer on call needs to understand: what does a correct charge look like? What does the invariant tell them? Where does the ledger state live? What is the deploy procedure and what safety gates exist?

The README, day notes, and ADRs answer these questions before the incident. The `/ec2-invariant` command and the deploy gate (`/check → /deploy-dry-run → manual deploy → /ec2-invariant`) are operational tools — but the README explains why they exist and what they prove. Without that explanation, the tools are just commands. With it, they are a documented safety system.

For a portfolio: the README is the signal that separates "I built a demo" from "I operated a real system." The difference is evidence. The invariant query returning 0 rows after every deploy is evidence. The three GitHub issues (INC-003, INC-004, INC-005) with forensic observations and resolution commits are evidence. The seven ADRs with explicit alternatives considered are evidence. The Week 1 hardening table with commit hashes is evidence. A reader who understands payments knows what all of this means.

### The broken pattern — what was demonstrated (and why Day 7 fixes it)

The old README described NovaPay as "a simulated production environment for a fictional fintech payment platform." It listed INC-001 (SSH disconnection) and INC-002 (disk exhaustion) in a table with no context, no resolution detail, no connection to what was actually built. It described the stack as "Docker + k3s (Phase 2)" and "Prometheus + Grafana (Phase 2)" — aspirational features, not built reality. A reader would have no idea what payment-api does, what the ledger is, why the invariant matters, or what the agentic workflow is.

This is the documentation debt pattern: the system advances, the docs stay at the initial placeholder. After six days of hardening work (bounded retries, goroutine lifecycle, context deadlines), the README still described a generic SRE lab.

**Observable problem:** anyone reading the repo would see six commits with commit messages referencing hardening work (D4, D5, D6) but no README that explains what any of it means. The evidence exists but is not surfaced.

### The correct pattern — what replaced it

The new README follows a case study structure:
- What was built → evidence (table with key properties)
- What was hardened → evidence (table with failure modes and commit hashes)
- How it was operated → evidence (agentic workflow, deploy gate, MCP integrations)
- Why each architectural choice was made → evidence (link to ADRs)

Every claim is backed by something verifiable: the invariant query (can be run), the commit hashes (can be checked), the GitHub issues (can be read), the ADRs (can be read). This is the same standard the codebase holds — the invariant query is the proof that the ledger is correct, and the README is the proof that the system was operated correctly.

### The failure cascade — what happens without documentation discipline

Without portfolio-quality documentation, the system accumulates knowledge in the builder's head that doesn't transfer:
- A future collaborator or interviewer reads the repo and sees a generic demo
- The reasoning behind ADR-004 (full jitter) is lost when someone asks "why not equal jitter?"
- The `/ec2-invariant` command exists but nobody knows what it proves or when to run it
- Week 2 planning requires reconstructing Week 1 context from scratch (slow, lossy)
- The LinkedIn article has no specific evidence to cite (the strongest articles are specific)

At scale: an operations team without documented procedures makes inconsistent decisions under pressure. The deploy gate exists — but if it's not in a README or runbook, the next engineer might skip `/deploy-dry-run` and push a broken binary to EC2.

## LinkedIn article notes

**Week 1 deep-dive — raw material for the article:**

**Strongest hook (one sentence):**
"I built a payments API that processed charges correctly for six days. Then I started counting goroutines, and found out it wasn't."

**The article structure (three-act):**
- Act 1 — The system looked healthy. `/healthz` responded in 11ms. Thread count: 9, stable. Every charge returned `approved`. The invariant: 0 rows. Everything correct.
- Act 2 — The three things that weren't. (1) The retry loop was amplifying PSP failures at scale — 10 charges became 23 PSP calls at 50% error rate, and the math gets exponential at 90%. (2) 15 charges created 15 zombie processes — the receipts were written, the files were correct, and there were 15 defunct processes nobody asked for. (3) goroutines climbed from 7 to 27 under 5 hung charges while `/healthz` kept saying `{"status":"ok"}`.
- Act 3 — Three defences, each structural (not just a fix). Full jitter so retries don't thundering-herd. In-process goroutine so no child processes exist. Context deadline so goroutines can't accumulate. After all three: the system degrades predictably — graceful 503s, stable goroutine count, 0 zombies, 0 unbalanced rows.

**The re-plan paragraph (for when the article is longer than expected):**
If the article runs long: cut Act 1 and lead with the goroutine story (most concrete, most surprising). The retry math and zombie section can become footnotes or a follow-up. The goroutine story has the best hook — "the service was up, the health check passed, zero charges completed" — and the best visual (7 → 27 goroutines, thread count: 9 throughout).

**Specific numbers to use:**
- Invariant: 0 rows throughout all of Week 1 — every payment balanced
- Retry: 10 charges → 23 PSP calls under 50% error rate (13 wasted)
- Zombies: 15 charges → 15 defunct processes, 1:1 exactly
- Goroutines: 7 baseline → 27 under 5 hung charges → back to 7 after fix
- Thread count (NLWP): 9 throughout the goroutine accumulation — OS sees nothing
- Hung charge resolution time: 5.01s each, all concurrent, wall time ~5.04s total
- Load test: 9/10 succeeded at PSP_ERROR_RATE=0.5 — bounded retry still working

**What NOT to make it about:**
- Kubernetes (not built yet)
- Prometheus/Grafana (Phase 2)
- The specific 5s/6s timeout values — they're examples, not the insight
- Generic "best practices" — lead with evidence, not advice

**The moment that resonates with a senior engineer:**
NLWP=9 while goroutines pile up. Any senior engineer who has debugged a goroutine leak has run `top` or `htop`, seen normal thread counts, concluded "nothing's wrong," and been wrong. The lesson: OS-level tools are blind to Go's runtime. `runtime.NumGoroutine()` in `/healthz` is not a nice-to-have — it is the only signal that catches this class of failure.
